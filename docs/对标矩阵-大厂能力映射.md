# 对标矩阵：大厂能力映射

本文档将当前数据平台能力与阿里云 DataWorks、腾讯云 WeData、字节跳动 DataLeap / BitSail 进行维度对标，并给出差距等级（P0/P1/P2）与升级路线。

---

## 1. 对标矩阵总表

| 能力维度 | 阿里云 DataWorks | 腾讯云 WeData | 字节 DataLeap | 当前平台现状 | 差距等级 | 升级计划 |
| --- | --- | --- | --- | --- | --- | --- |
| 数据采集集成 | 多源批流一体、CDC | 多引擎（EMR/DLC 等） | BitSail 插件化 | Flink + BulkLoad 模板 | P1 | 扩展连接器与 CDC 能力，抽象采集插件层 |
| 流程编排调度 | DAG 可视化、周期调度 | WeData 工作流 | 调度引擎 / Gödel | 6 步顺序编排 | P0 | 引入 DAG 与周期/依赖调度，可视化编排 |
| 参数配置管理 | 环境变量、连接管理 | 参数模板、版本 | 配置中心化 | 模板 + AI 推荐 | P1 | 连接与密钥治理、参数版本与发布流 |
| 质量监控 | DQC 规则引擎、阻断 | 76 系统规则、字段级监控 | 基线监控、全链路 | 基础事件 + 异常检测 | P0 | 规则引擎、字段级校验、失败阻断策略 |
| 智能告警 | 智能基线、预测告警 | 告警阻断下游 | 基线监控覆盖 80%+ | 严重度分级 | P0 | 基线/预测、下游联动、降噪与收敛 |
| 根因分析 | 血缘追踪 | 全链路血缘 | 依赖链分析 | 无 | P0 | 血缘元数据、依赖图、故障定位视图 |
| 知识管理 | 数据地图 | 数据知识库 / 资产 | 数据发现 | 知识库 + 关键词检索 | P1 | 资产目录、标签、与任务/表关联 |

| 成本分析 | 资源优化 | 冷热分层、成本治理 | 资源利用率优化 | 无 | P2 | 资源用量采集、成本归因与优化建议 |

## 2. P0 优先升级项详解

### 2.1 流程编排调度

| 项目 | 说明 |
| --- | --- |
| **当前差距** | 仅支持固定 6 步线性顺序，缺少 DAG、分支合并、周期与跨任务依赖调度，无法对标大厂可视化工作流与生产级调度引擎。 |
| **对标目标** | 达到 WeData / DataWorks 级别的 DAG 编排与周期调度能力：任务依赖、重跑策略、可视化编辑与运行态可观测。 |
| **升级要点** | 定义任务节点与边模型；实现调度器（依赖解析、就绪队列、失败重试与跳过策略）；前端 DAG 画布（或分阶段先支持依赖列表 + 拓扑排序）；与现有 Flink/BulkLoad 步骤对接为可插拔节点类型。 |
| **涉及文件** | `backend/routers/flows.py`、`backend/models.py`、`frontend/src/pages/FlowManager.jsx`、`frontend/src/api.js`（及后续新增的调度服务/Worker 模块路径） |

### 2.2 质量监控

| 项目 | 说明 |
| --- | --- |
| **当前差距** | 仅有基础事件与简单异常检测，缺少可配置规则引擎、字段级规则、系统级规则模板及失败时对下游的强阻断能力。 |
| **对标目标** | 对齐 DQC / WeData 规则体系：规则模板、阈值与自定义 SQL、调度挂钩、质检报告与阻断策略。 |
| **升级要点** | 规则元数据与执行引擎（批/微批）；结果落库与历史对比；与编排联动（质检失败是否阻塞下游）；字段级与表级规则分层。 |
| **涉及文件** | `backend/routers/monitor.py`、`backend/models.py`、`frontend/src/pages/Monitor.jsx`、`frontend/src/api.js` |

### 2.3 智能告警

| 项目 | 说明 |
| --- | --- |
| **当前差距** | 告警以严重度分级为主，缺少智能基线、趋势预测、告警与任务/血缘联动及下游自动阻断或降级策略。 |
| **对标目标** | 接近 DataWorks 智能基线与 DataLeap 高覆盖基线思路：动态阈值、预测性告警、告警路由与处置闭环。 |
| **升级要点** | 指标基线学习与异常评分；告警策略（静默窗、合并、升级）；与调度/质检状态联动；可选对接企业 IM/Webhook。 |
| **涉及文件** | `backend/routers/monitor.py`、`backend/services/ai_engine.py`（基线/预测可逐步接入）、`frontend/src/pages/Monitor.jsx`、`frontend/src/pages/Dashboard.jsx` |

### 2.4 根因分析

| 项目 | 说明 |
| --- | --- |
| **当前差距** | 平台侧无血缘与依赖链分析，故障时需人工排查，无法对标全链路血缘与依赖链定位。 |
| **对标目标** | 具备表级/任务级血缘与依赖图，支持从告警或失败任务一键展开上游影响面与可疑根因节点。 |
| **升级要点** | 采集 SQL/任务配置解析血缘（或对接元数据中心）；存储图结构（节点/边）；API 与前端依赖图/时间线视图；与监控告警跳转打通。 |
| **涉及文件** | 新增建议：`backend/routers/lineage.py`（或并入 `flows.py`）、`backend/models.py`、新建 `frontend/src/pages/Lineage.jsx`（或扩展 `Dashboard.jsx`）、`main.py` 路由注册 |

## 3. 实施优先级路线

### Phase 1：P0（约 8～14 周，视团队规模浮动）

- **编排调度**：DAG 数据模型 + 调度内核 MVP，再迭代可视化画布与周期调度。
- **质量监控**：规则引擎与若干内置规则，与编排阻断打通。
- **智能告警**：基线/阈值策略、告警合并与路由；与监控事件、任务状态关联。
- **根因分析**：血缘元数据与只读依赖图 MVP，告警跳转至血缘视图。

### Phase 2：P1（约 10～16 周）

- **采集集成**：CDC、多源连接器与插件化边界。
- **参数配置**：连接管理、参数版本与发布审批。
- **知识管理**：资产目录、标签与任务/表关联检索。
- **设计系统**：选定 Ant Design / TDesign / Arco 之一并渐进迁移关键页面。
- **权限审计**：RBAC、资源级权限、操作审计日志。

### Phase 3：P2（约 8～12 周，可与 P1 部分并行规划）

- **AI Copilot**：在 SQL、配置生成、排障摘要等场景落地，与 `ai_engine` 能力结合。
- **成本分析**：资源用量采集、成本归因模型与优化建议看板。

> **说明**：上述周数为粗估，需按人力、合规与外部系统对接情况滚动调整；P0 项建议严格串并行规划，避免调度与质检/告警割裂实现。
